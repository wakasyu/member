## プロジェクトの概要

- 若衆（伝統的な青年団）のメンバー専用サイト。GAS/スプレッドシート運用をやめて、
  素のHTML/CSS/JS + Supabase（Postgres, Auth, Storage, Edge Functions）で作った。
- ビルドツール無し。`index.html` を直接ブラウザで開くか、GitHub Pagesで配信。

## できていること（機能一覧）

- ログイン（メール/パスワード）、パスワード再設定メール
- ログイン後の「トップ」ページ：Supabase Storageの非公開バケット
  （`top-photos`）から写真を取得し全画面背景でスライドショー表示。
  ヘッダー・タブ・タイトルは写真の上にガラス風パネルで重ねる。
  写真は管理画面「トップ写真」タブからアップロード/削除（管理者のみ書込み、
  ログイン済みなら誰でも閲覧）
- 予定一覧：日ごとに見出し、コンパクトな1行レイアウト（空の項目は非表示）、
  出欠人数はインラインバッジ、日程調整リンク/共有文コピーは見出し行の右側
  （狭い画面では自然に折り返す）
- 日程調整回答：ログイン中の本人として回答（名前選択は無し）。
  管理者だけ「代理入力するメンバー」を選んで他人の回答を編集可能。
  回答期限を過ぎると一般メンバーは編集不可（UIとRLS両方で強制）。
  回答の取り消し（未回答に戻す）も可能
- 管理画面：予定/メンバーのCRUD、削除は論理削除（予定は復元可）、
  分類・理由カテゴリの選択肢管理、操作ログ（実行者名も記録）、
  トップ写真管理、メンバー表示順の並び替え（▲▼ボタン）
- メンバー一覧・予定一覧の管理者用テーブルは幅700px以下でカード表示に自動切替
  （横スクロール不要）
- メール通知（Resend経由）：回答があるたびではなく、対象メンバー全員の
  回答が揃った時だけ管理者に通知
- リアルタイム反映（Supabase Realtime）

## データベース・インフラの状態

- `supabase-schema.sql` は繰り返し実行安全だが、**このファイルを更新したら
  Supabase側で実際にSQL Editorで再実行してもらう必要がある**（自動デプロイの
  仕組みは無い）。直近の変更で `members.sort_order` 列を追加したが、
  ユーザーが実行済みか最終確認が取れていない（2026-07-13時点で一度
  "column does not exist" エラーが出て、実行を依頼した）。
- Storageバケット `top-photos`：非公開。RLSで「認証済みなら閲覧可、
  書き込みは管理者のみ」。
- Edge Function `notify-answer`：`supabase functions deploy notify-answer --no-verify-jwt`
  でデプロイ済み。secretsに `RESEND_API_KEY` / `ADMIN_NOTIFY_EMAIL` / `NOTIFY_FROM_EMAIL`
  / `WEBHOOK_SECRET` を設定済み。`answers`テーブルへのWebhookトリガー
  （`notify_answer_change`）も設定済み。
- Resendは無料枠でドメイン未認証のため、`ADMIN_NOTIFY_EMAIL`に設定した
  アカウント登録メール以外には送信できない。他の管理者にはGmailの
  転送フィルタで届くよう設定済み（サイト側の実装ではなく利用者側の設定）。
- Supabase Personal Access TokenやCLIログインは作業のたびに一時的に
  発行してもらい、作業後は失効させる運用にしている（常時保持しない）。

  `profiles`行の紐付け）は今もダッシュボードからの手動作業のまま
  （アプリ内に招待機能は無い）

## 作業の進め方（このプロジェクトでの申し合わせ）

- 変更は都度 `git commit` → `git push` している（GitHub CLIでブラウザ認証、
  トークンは使い終わったら失効させる運用）
- SupabaseのSQL/Storage/Edge Functionをこちらから直接操作する場合は、
  ユーザーにPersonal Access Tokenを都度発行してもらい、作業後は
  ダッシュボードから削除してもらっている
- 見た目の変更は可能な範囲でこのブラウザプレビュー環境で確認してから
  pushしているが、この環境のスクリーンショット機能は時々タイムアウト/
  幅の大きいウィンドウを正しく撮れないことがある。その場合は
  `getComputedStyle`や`getBoundingClientRect`などJS経由の数値検証で
  代替している
- ユーザーは「プロンプトを作って実行して」という形で、要件を明確化した
  上で実装からGitHubへの反映まで一気に任せるスタイルを好む
